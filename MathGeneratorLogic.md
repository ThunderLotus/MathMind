# 小学生口算题出题核心逻辑与算法设计

本文档提炼了当前系统中基于配置参数 (`AppSettings`) 动态生成四则混合运算题目的核心抽象逻辑。该逻辑通过**抽象语法树(AST)构建**结合**蒙特卡洛随机重试**来实现复杂约束下的生成，非常适合被迁移、复刻至 Java、Python、C# 等任意后端语言或平台。

## 1. 核心架构：抽象语法树 (AST)

为了灵活支持多步运算以及正确的括号规则（优先级处理），题目生成没有采用简单的字符串拼接，而是构建了**二叉树（AST）**。

- **节点定义 (`ExprNode`)**：
  - **叶子节点** (类型为 `num`): 只包含数值（建议统一使用分数表示，整数为分母为1的分数）和呈现格式（真分数、假分数、带分数、整数）。
  - **非叶子节点** (类型为 `op`): 包含算符(`+`, `-`, `×`, `÷`)，以及左子节点(`left`)和右子节点(`right`)。

- **底层数据结构：分数计算类 (`Frac`)**：
  为避免浮点数精度丢失，系统内所有的数值一律封装为 `(分子 Numerator, 分母 Denominator)` 结构，并在所有加减乘除运算结束后自动计算最大公约数(GCD)进行约分。如果是纯整数运算，则分母恒为1。

## 2. 核心主流程：重试机制 (Generator Loop)

由于用户可能组合出极其苛刻的约束条件（如：结果不能为负数、必须限制在100以内、必须包含括号、整数除法必须除尽等），正向直接生成符合所有要求的数据难度极大。因此系统采用了**生成->验证->重试**的反馈循环机制。

**主流程步骤**：
1. **提取同分母设置**：如果配置需要生成同分母分数，先在全局随机确定一个分母 `commonDenom`。
2. **进入重试循环 (如 500 次)**：
   1. **生成语法树 (`randomAST`)**：根据指定的运算次数 `opCount` 随机生成一棵二叉树。
   2. **树求值 (`evaluate`)**：对生成的树使用 `Frac` 类进行真实计算。如果计算过程中出现除以0，或整数模式下除不尽（分母不为1），求值直接中断返回 `null`。
   3. **结果校验**：
      - 检查是否有有效结果。
      - 检查结果是否为负数（小学生题目通常不允许结果小于0）。
      - 检查结果是否超出最大范围限制（如 `resultMax <= 100`）。
   4. **结构校验 (括号验证)**：
      - 根据树结构和运算符优先级，计算这棵树如果转成文本会产生多少对括号。
      - 检查是否符合用户的括号配置要求（`hasParens`：仅一次包含，或者不包含）。
   5. **成功返回**：如果全满足，则跳出循环，输出该题。
   6. **失败继续**：如果不满足，抛弃这棵树，重试下一次循环。
3. **循环结束未找到**：向外抛出失败标识。

## 3. AST 子模块：树生成逻辑 (`randomAST`)

采用递归拆分法分配“剩余操作符数量”，保证树形结构的随机性。

- **输入**: 当前剩余的操作符配额 `opCount`。
- **基线条件**: 若 `opCount == 0`，代表此处应该产生一个具体的数字，调用**数字生成模块**，返回叶子节点。
- **递归分配**:
  1. 随机选出一个运算符分配给当前节点。
  2. 剩余还需要分配的运算符数量为 `rem = opCount - 1`。
  3. 取一个随机数 `leftops` 在 `[0, rem]` 之间。
  4. 算得 `rightops = rem - leftops`。
  5. 递归产生左子树：传入 `leftops`；递归产生右子树：传入 `rightops`。

## 4. AST 子模块：数字生成逻辑 (`generateValNode`)

当树到达叶子节点时，根据全局配置动态生成具体的数值。

**分支 1：生成整数**
- 根据参数 (`num1Digit`, `num2Digit`) 确定范围，如 1~9, 10~99, 或 1~99。
- 随机生成该范围内的整数，以分母为 1 的形式存入 `Frac`。

**分支 2：生成分数 (含带分数)**
- **分母生成**：如果是“同分母”模式，则使用全局传进来的 `commonDenom`；否则根据分母范围配置(1~9 或 10~99)重新随机。当遇到计算规则需避开分母为1时，强制改为2。
- **格式选取**：综合(`allowProper`, `allowImproper`, `allowMixed`)从可用项中随机选出一个格式。
- **分子生成**：
  - _真分数 (proper)_: 分子 `n` 在 `[1, 分母-1]` 之间随机（如允许分子为0，则底限为0）。
  - _假分数 (improper)_: 分子 `n` 在 `[分母, 分母*2]` 之间取值，防止数字膨胀过大。
  - _带分数 (mixed)_: 先随机整数部分 `W` (如1~5)，再随机真分数部分 `n_proper < 分母`，最终转为普通假分数存入系统 `(W * 分母) + n_proper`，但节点上标记展示格式为 `mixed`。

## 5. 辅助模块：括号判定法则 (`checkNeedsParens`)

该模块用于判断 AST 中某个非叶子节点在渲染输出时，是否需要加上括号 `()`。

- **基本逻辑：**比较当前操作符 `cOp` 与其父节点操作符 `pOp` 的优先级（乘除=2，加减=1）。
- **判定规则：**
  1. 如果 `cOp.level < pOp.level`（例如 `+` 是 `×` 的子节点），**必须加括号**。
  2. 如果 `cOp.level == pOp.level` 并且当前节点是父节点的**右子节点**：
     - 若 `pOp` 是减号 `-`，且 `cOp` 是加号 `+` 或减号 `-` （例如 `A - (B + C)`），**必须加括号**。
     - 若 `pOp` 是除号 `÷` （例如 `A ÷ (B × C)`），**必须加括号**。
  3. 其他情况**无需加括号**（根据结合律从左向右计算即可）。

**迁移实施建议**：
在迁移至 Java 或 C# 等强类型语言时，建议：
1. 实现一个高精度的 `Fraction` 实体类，重写 `add`, `sub`, `multiply`, `divide` 方法并保证在每次操作后调用 `gcd`(最大公约数) 化简。
2. 定义面向对象的 AST（例如抽象接口 `ExpressionNode` 派生 `NumberNode` 与 `OperatorNode`）。
3. 渲染视图层（UI 或 打印）应与此数据生成引擎解耦，引擎最终只返回序列化好的计算式（包含括号及结果的文本或对象树）。
